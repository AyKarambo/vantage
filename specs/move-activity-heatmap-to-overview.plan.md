# Tech Plan: Move Activity Heatmap to Overview

**Slug:** `move-activity-heatmap-to-overview` · **Spec:** `specs/move-activity-heatmap-to-overview.spec.md` · **Source:** [issue #116](https://github.com/AyKarambo/vantage/issues/116)

## Architecture & Approach

Pure renderer relocation — no new component, no data changes. `calendarHeatmap` (`renderer/src/components/primitives/stats.ts:69-95`) is untouched; only its call site moves from `trends()` to `overview()`'s `bottomRow()`.

**Key fact from research that simplifies this plan:** the heatmap window is a fixed 35 days (`calendar(games, 35)`, `src/core/analytics/session.ts:51`, called unconditionally from `computeDashboard()` at `src/core/dashboardData.ts:124`) — it does **not** scale with the active date-range filter. With `grid-auto-flow: column` and 13px cells (`.heatmap`, `renderer/styles/components.css:652-658`), 35 days always lays out as **5–6 columns**, ~78–93px of grid plus its legend (`.heatmap-legend`, `components.css:665-669`) in a `flex-wrap: wrap` wrapper (`.heatmap-wrap`, `components.css:646-651`). This is a narrow, near-fixed-width card — it drops into the existing two-card flex row as an equal third with no CSS changes, no horizontal scroll, and no weighted-flex sizing needed. The "will it fit" open question from the spec is resolved: it fits cleanly.

1. **`overview.ts`** — `bottomRow(ctx)` (`renderer/src/views/overview.ts:185-201`) currently returns `h('div', { class: 'overview-bottom' }, mental, readinessCard(ctx))`, a 2-child flex row (`.overview-bottom { display: flex; gap: 12px; }`, `renderer/styles/app.css:311-314`). Add a third card built the same way the removed Trends card built it — `card({ title: 'Activity', sub: 'games/day · colour = winrate · click a day to open its matches', style: { flex: '1' } }, calendarHeatmap(d.calendar, (date) => ctx.navigate('matches', { day: date })))` — as a new `activityCard(d)` helper (mirrors the existing `mental`/`readinessCard` structure), appended as the third argument. `readinessCard` already returns `null` safely when disabled (`dom.ts:64-69` confirms `h()`/`append()` drop `null` children via `children.flat()` + `if (child == null ...) continue`), so the row holds 2 or 3 cards without any conditional logic change.
2. **`trends.ts`** — delete the `card({ title: 'Activity', ... }, calendarHeatmap(...))` block (`renderer/src/views/trends.ts:37-38`) and the `calendarHeatmap` import from the `components/primitives` import line (`trends.ts:7`); keep `card`, `emptyState`, `statBox` (still used by other cards in the file).
3. **Specs** — remove the Activity bullet (`screen-trends.spec.md:16`) and its AC line (`screen-trends.spec.md:33`); add an equivalent In-Scope bullet + AC to `screen-overview.spec.md`'s bottom-row description (§ In-Scope, currently line 19) and AC list (currently lines 35-43).
4. **README.md** — Definition of Done requires docs updates when user-visible behavior/placement changes:
   - Line 63-69 (Trends bullet): drop the `, an activity heatmap,` clause from the sentence.
   - Line 23-24 (Overview bullet): append the heatmap, e.g. `and a mental snapshot.` → `a mental snapshot, and an activity heatmap (games/day, click a day to open its matches).`

No changes to `src/shared/contract/`, `src/core/`, `stats.ts`, or any CSS file — confirmed sufficient by the column-width research above.

## Test Strategy

- No renderer view unit tests exist in this codebase (confirmed: `test/` covers `src/core/` only, per project convention) and this change touches no `src/core/` logic — so Definition of Done #3 ("new or changed pure logic under `src/core/` ships with unit tests") doesn't apply; no new test files.
- `npm run typecheck` (main + renderer) must stay clean — the change is straightforward JSX-less DOM composition matching existing patterns exactly, low type-risk.
- `npm test` must stay green unchanged (no core logic touched).
- Manual/visual verification via the preview harness (`npm run preview` or the standalone preview server per this repo's worktree convention):
  1. Overview renders three cards in the bottom row — Mental, Readiness, Activity — at normal window width, no overflow/wrapping oddities.
  2. Toggle Readiness off (Settings) → bottom row still renders correctly with 2 cards (Mental + Activity), no gap/layout artifact from the missing third.
  3. Click a heatmap day cell that has games → navigates to Matches filtered to that day (same as today's Trends behavior).
  4. Trends renders with no Activity card, heatmap, or leftover teaser anywhere on the screen.
  5. `grep -n calendarHeatmap renderer/src/views/trends.ts` returns nothing (import fully removed).

## Risks & Alternatives

- **Risk (low):** three cards in a previously two-card row could look cramped on narrow windows. Mitigated by the heatmap being the narrowest of the three cards by a wide margin (~93px + legend vs. Mental/Readiness's fuller content), and `.heatmap-wrap`'s existing `flex-wrap: wrap` already lets the legend drop below the grid if its column gets tight — no CSS change anticipated, but confirm visually during implementation (step 1 of Test Strategy).
- **Alternative (rejected in spec review):** a separate full-width row below `bottomRow`, matching `scatterCard`'s treatment — rejected by the user as leaving most of the row as empty space given the heatmap's small footprint.
- **Alternative (rejected in spec review):** leave a "moved to Overview" teaser/stub card on Trends — rejected per the issue's own recommendation (confirmed by user); the persistent sidebar nav already makes Overview one click away.
- **Doc-drift risk (low):** the README bullet edits are prose-only; kept terse and consistent with the existing bullet style to avoid introducing new claims.
